Edge-Based Document AI: Architectural Strategies for In-Browser Structured Invoice Extraction
The rapid advancement of web-runtime acceleration has unlocked the feasibility of executing complex deep learning tasks entirely client-side, running locally within the end-user’s browser via WebAssembly (WASM) or WebGPU through the ONNX Runtime1. In-browser processing guarantees absolute data privacy, eliminates per-token API billing, and allows for offline functionality1. However, parsing unstructured, arbitrary invoice documents to consistently extract high-fidelity fields—such as invoice numbers, transaction dates, vendor/buyer entities, line items, and financial totals—presents an engineering challenge on edge devices.
General vision-language models like SmolVLM-500M-Instruct often fail on structured document tasks because they lack layout-aware inductive biases, resulting in coordinate drifting, key-value mismatching, and numerical hallucinations2. This report evaluates document-specialized architectures, compares single-stage vision-language models with decoupled two-stage pipelines, and analyzes cloistered runtime execution strategies under strict browser constraints1.

	Candidate Trade-Off Analysis
Among these options, Florence-2 and Donut are the only generative models that fit within a 500 MB browser budget while maintaining native compatibility with Transformers.js3.
LayoutLMv3 is an encoder-only architecture. This means it cannot generate text directly. Instead, it classifies tokens that have already been extracted and mapped by an external OCR engine. This limitation makes it less flexible for zero-shot extraction on varied invoice layouts.
TrOCR is highly accurate for transcribing single lines of printed or handwritten text15, but it lacks the structural awareness needed to parse multi-column document layouts15.
Models like Qwen2-VL-2B and Phi-3.5-vision offer strong accuracy but are too large for standard web runtimes28. Their model files (often exceeding 1.5 GB even with quantization) lead to high network latency, large memory usage, and frequent browser tab crashes32.
Browser-Deployable OCR and Extraction Pipelines: Architectural Trade-Offs
When designing an in-browser extraction engine, developers must choose between a Single-Stage Multimodal Model (VLM) or a Decoupled Two-Stage Pipeline.

The Decoupled Two-Stage Pipeline
The decoupled pipeline separates the extraction process into two distinct stages:
1. First Stage (Text Extraction): A small, specialized optical character recognition (OCR) engine extracts text characters and their exact bounding box coordinates from the image34.
2. Second Stage (Information Extraction): A text-only Small Language Model (SLM) processes the extracted text, using the layout coordinates to parse and organize the data into the target fields6.
This approach can be implemented using either Tesseract.js or PaddleOCR for the first-stage text extraction:
* Tesseract.js: A port of the classic Tesseract engine36. It relies on an older LSTM architecture and requires downloading language-specific files (such as nor for Norwegian)37. Tesseract.js is prone to errors on dense, multi-column invoice tables, often misaligning columns or misreading low-resolution text37.
* PaddleOCR (via ppu-paddle-ocr): A modern, highly optimized OCR engine34. It runs PP-OCRv5/v6 models natively in the browser using ONNX Runtime Web37. The combined text detection and recognition models are small (under 15 MB) and run quickly on consumer CPUs or WebAssembly runtimes37. This engine delivers high accuracy (achieving up to 99.22% on receipt benchmarks) and handles complex multi-column structures and tabular layouts with minimal errors37.
Once the text and layout coordinates are extracted, they are formatted into a structured text payload and passed to a small, text-only model like Qwen2.5-0.5B-Instruct or SmolLM2-135M-Instruct6. These text-only models are small (under 350 MB in 4-bit quantization) and highly optimized for WebGPU execution6.

	Edge Bottlenecks: Vision Encoding vs. Autoregressive Decoding
Analyzing edge execution paths reveals an asymmetrical performance bottleneck. In single-stage VLMs, the vision encoder processes high-resolution image matrices to generate visual hidden states41. While this visual pass is computationally heavy, it runs only once per image41.
The primary latency bottleneck is the text decoder41. The decoder must run autoregressively, performing a full forward pass to generate each individual token41. On edge hardware using WebAssembly or low-power WebGPU configurations, this continuous text generation can slow down performance15.
Because VLMs represent images using hundreds of visual tokens, the decoder must maintain a large key-value (KV) cache31. This increases memory usage and slows down execution on resource-constrained devices15.
Detailed Evaluation of the Donut Architecture
The Document Understanding Transformer (Donut) was developed by NAVER Clova as an OCR-free system for document parsing42. It processes document images directly using an end-to-end transformer architecture, reading visual features and generating structured text without an intermediate OCR step2.
ONNX Availability and Size Under Quantization
Donut models are available in the ONNX format and can be loaded in the browser using Transformers.js11. For example, the Xenova/donut-base-finetuned-docvqa checkpoint is natively supported under the document-question-answering task pipeline3.
The model's file sizes vary depending on the quantization level applied to its two main components:
1. Visual Encoder (Swin Transformer): The FP32 version is ~150 MB. When converted to 4-bit quantization (such as the bnb4 format), its footprint drops to 57.1 MB14.
2. Autoregressive Decoder (BART): The unquantized FP32 model is 505.1 MB11. Applying an 8-bit quantization (q8) reduces this component to 130 MB, bringing the combined model size to around 187 MB12.
While these sizes fit within browser storage budgets, compiling Donut's large sequence-to-sequence decoder in WebAssembly or WebGPU can cause issues15. The complex ONNX graphs often trigger compilation timeouts or out-of-memory errors on devices with limited RAM15.
Structural Reliability on European and Multilingual Invoices
Although Donut works well on standard, pre-trained datasets like DocVQA or SROIE13, its OCR-free architecture has limitations when processing unseen European invoices.
Donut relies on its decoder's vocabulary to transcribe text directly from visual features2. Without a deterministic OCR step to extract raw characters, the model must read and parse layout structures simultaneously2. When processing unfamiliar layouts or non-English documents, this can lead to several errors:
* Transcription Hallucinations: Because Donut does not use a direct character recognition step, it can misread critical financial figures (such as total amounts or invoice numbers) if the visual quality is low or the layout is complex2.
* Diacritic Sensitivity: European languages, such as Norwegian, use unique diacritical marks (e.g., æ, ø, å)49. Because Donut is trained primarily on English-centric document datasets, its visual encoder often struggles to resolve these characters, leading to spelling errors in vendor names and addresses.
* Layout Drift: If an invoice format differs from the structures in the training data, Donut can lose track of document hierarchy. This causes it to miss key-value associations or fail to output the correct target schema tags2.
Managing Practical Browser Constraints and Runtime Acceleration
Deploying neural network pipelines directly in the browser requires managing strict resource limitations1. To ensure a responsive user experience, the system must balance model download sizes with runtime execution speed1.
Optimizing Engine Configurations: WebGPU and WebAssembly
To achieve acceptable performance on client devices, developers should prioritize the WebGPU execution provider (device: "webgpu") over WebAssembly (WASM)1. WebGPU provides direct access to GPU hardware acceleration, delivering up to 100x faster execution than CPU-bound WASM runtimes1.
However, because some model layers are highly sensitive to quantization, a mixed-precision execution strategy is recommended:






JavaScript
// Mixed-precision modular instantiation configuration for Florence-2 in Transformers.js
const model = await Florence2ForConditionalGeneration.from_pretrained("onnx-community/Florence-2-base-ft", {
 dtype: {
   embed_tokens: "fp16",       // Preserves coordinate precision for layout mapping
   vision_encoder: "fp16",     // Preserves detail in visual embeddings to avoid character loss
   encoder_model: "q4",        // Compresses the text encoder to save memory
   decoder_model_merged: "q4"  // Compresses the autoregressive decoder to speed up generation
 },
 device: "webgpu"              // Enables high-speed WebGPU acceleration
});

Using this mixed-precision configuration with onnx-community/Florence-2-base-ft balances memory usage and performance6:
* Embedding and Vision Towers: Keeping these components in fp16 prevents visual detail loss6.
* Autoregressive Decoders: Quantizing these layers to 4-bit (q4 or q4f16) reduces model size and speeds up token generation6.
Managing Model Sizes and Download Budgets
To keep initial download times reasonable, the entire model footprint should fit within a 500 MB browser budget3.
When using Florence-2, the combined size of the mixed-precision files is highly optimized7:
* vision_encoder_fp16.onnx is 367 MB7.
* embed_tokens_fp16.onnx is 78.8 MB7.
* encoder_model_q4.onnx is 30.1 MB7.
* decoder_model_merged_q4.onnx is 64.4 MB7.
This configuration requires a total download of ~540 MB7. While slightly over the preferred target, the browser natively caches these files in IndexedDB on the first run, making subsequent loads instantaneous1.
Alternatively, if a smaller footprint is required, the vision encoder can be quantized to 4-bit (bnb4), reducing its size to 76 MB and bringing the total download budget down to ~249 MB7. This lower footprint makes the application much more accessible on slow network connections3.
Ensuring Structured JSON Schema Outputs
A common issue when running text generation models on the edge is output consistency. In-browser models can easily output malformed JSON, break schemas, or generate endless repetitive loops46.
To solve this, developers can integrate llguidance (transformers-llguidance) directly into the pipeline54. This library compiles a target JSON schema into a finite state machine (FSM) at runtime55.
During token generation, the FSM acts as a logit processor, masking out tokens that would violate the schema55. This forces the model to generate syntactically valid JSON that perfectly matches the required fields55.

For a client-side invoice scanner running entirely in-browser under a 500 MB memory budget, the decoupled two-stage pipeline is the most reliable and performant architecture.

Why the Decoupled Pipeline Succeeds on the Edge
1. Guaranteed Data Integrity: Financial applications require precise character transcription2. Separating text detection from semantic parsing ensures that the language model cannot hallucinate or invent numbers2. The system only extracts text blocks directly identified by the OCR stage2.
2. Robust Multilingual Handling: European invoices, particularly from Nordic regions, include special characters like æ, ø, and å49. While visual document models often misidentify these letters, multilingual OCR engines (such as PP-OCRv6) support them natively35, ensuring accurate vendor names and addresses.
3. Low Latency on Edge Hardware: Processing images through a visual transformer's cross-attention layers on the edge is computationally expensive and slow2. By decoupling the stages, the lightweight OCR model runs in milliseconds37. The text-only language model then processes a small, clean token payload, resulting in a fast, responsive user interface.
Recommended Implementation Details
For developers building this pipeline, the following configuration balances accuracy, speed, and size:
* First Stage (OCR): Use ppu-paddle-ocr to run optimized PP-OCRv6 graphs via ONNX Runtime Web37. This stage detects characters and outputs their coordinates35.
* Second Stage (Semantic Parsing): Use Qwen2.5-0.5B-Instruct quantized to 4-bit (q4) via Transformers.js6. Use the transformers-llguidance library to enforce the target JSON schema, ensuring consistent and valid outputs54.
